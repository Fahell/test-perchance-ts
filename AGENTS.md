## Project Overview

This project modularizes TypeScript for use in Perchance (https://perchance.org), a procedural generation platform. Each module in `src/modules/` tests a specific plugin or functionality within the Perchance ecosystem.

## Project Structure

- `src/main.ts` — Entry point (imports and initializes modules)
- `src/perchance-bridge.ts` — Safe bridge to Perchance root API/lists
- `src/constants.ts` — Global constants (version, CDN)
- `src/modules/*.ts` — Independent modules (renderer, AI text test, Three.js test)
- `src/modules/types.ts` — Shared TypeScript types
- `dist/main.bundle.js` — Single bundle generated by Vite (committed for CDN)
- `for-perchance.html` — HTML Panel for copy/paste into Perchance
- `for-perchance-list-panel.txt` — List Panel for copy/paste into Perchance
- `tests/*.test.ts` — Unit tests with Vitest
- `scripts/sync-version.cjs` — Version synchronization script
- `.release-config.json` — Configuration for agent-release automation

## Coding Rules

- No `var`, use `const`/`let`
- No `console.log` in production (use logger)
- No `==`, use `===`
- No `any` type (use proper TypeScript types)
- ESLint + Prettier mandatory
- Always use `.js` extension in relative imports for ESM compatibility: `import {...} from './module.js'`

## ⚠️ Critical Guidelines

### Known Errors to Avoid

1. Avoid unnecessary `try/catch` in async functions
2. Always use relative imports with `.js` extension: `import {...} from './module.js'`
3. Do not ignore `dist/` in `.gitignore` — the bundle must be committed for the CDN to work

### Mandatory Workflow
- When planning: Always use Websearch before implementing new features, fixes, edits and planning. Analyse the request and plan using Websearch. Websearch is your biggest and best tool.
- Use feature branches: `feature/auth-refactor`
- Never commit directly to `main`
- Use `release X.Y.Z` for automated release (update, sync, build, commit, tag, push, PR, merge, GitHub release). See README.md.

## 🎯 Prompt Symbols Reference

The shell prompt provides structured context to avoid unnecessary commands. Each symbol has fixed semantic meaning:

### Format
```
[project@version] cwd git:branch +N ~N ?N ↑N ↓N *N STATE !code $
```

### Symbol Glossary

#### Project Context
- `[name@version]` — Project name and version (detected from package.json)

#### Git Status (in order of appearance)
- `git:branch` — Current branch name
- `+N` — N files staged for commit (ready to commit)
- `~N` — N files modified but not staged (needs `git add`)
- `?N` — N untracked files (new files not in git)
- `↑N` — N commits ahead of remote (ready to push)
- `↓N` — N commits behind remote (needs `git pull`)
- `*N` — N stashes saved (use `git stash list` to see)
- `MERGE` — Merge in progress (resolve conflicts)
- `REBASE` — Rebase in progress (resolve conflicts or continue)
- `CHERRY` — Cherry-pick in progress

#### System
- `!N` — Last command exit code (shown only when ≠ 0, indicates error)
- `$` — Normal user prompt
- `#` — Root user prompt

### Examples

**Working tree clean:**
```
[test-perchance-ts@1.0.0] main git:main $
```
→ No changes, branch up-to-date, can start work immediately.

**Changes pending:**
```
[test-perchance-ts@1.0.0] src git:feat-bridge +2 ~1 ?1 $
```
→ 2 files staged (ready to commit), 1 modified (needs staging), 1 untracked (new file).

**Branch behind remote:**
```
[test-perchance-ts@1.0.0] main git:main ↓5 $
```
→ Branch is 5 commits behind remote. Run `git pull --rebase` before working.

**Ready to push:**
```
[test-perchance-ts@1.0.0] main git:main ↑3 $
```
→ 3 commits ahead of remote. Run `git push` to publish.

**Merge conflict:**
```
[test-perchance-ts@1.0.0] src git:main MERGE +2 ~3 !1 $
```
→ Merge in progress, last command failed (exit code 1). Resolve conflicts and commit.

**With stashes:**
```
[test-perchance-ts@1.0.0] main git:dev *2 $
```
→ 2 stashes saved. Be careful when switching branches.

**Outside project:**
```
projects git:main $
```
→ No project detected (not in a Node/Python/Rust/etc project directory).

### Actions Based on Prompt

When you see:
- `+N` → Can commit: `git commit -m "message"`
- `~N` → Stage first: `git add .` then commit
- `?N` → Decide if files should be tracked or added to .gitignore
- `↑N` → Push changes: `git push`
- `↓N` → Pull first: `git pull --rebase`
- `*N` → Check stashes: `git stash list`
- `MERGE/REBASE/CHERRY` → Resolve conflicts and complete operation
- `!N` → Previous command failed, check what went wrong

## 🛠️ Environment Features

### Optimized Shell Configuration

The environment is optimized for AI agents with:

- **Clean outputs**: No colors, no pagers, single column for lists
- **Context injection**: Prompt provides cwd, git branch, and project context
- **Smart aliases**: `git`, `eza`, `bat`, `grep` configured for non-interactive use
- **No hang protection**: `less` aliased to `cat`, pagers disabled
- **Modular config**: Loaded from `~/.zshrc.d/*.zsh`

### Available Tools

- **Release automation**: `release X.Y.Z` (global command from agent-release)
- **Version sync**: Automatic via `sync-version.cjs` hook
- **Agent session manager**: `ag-w` (for human observation), `agent-session list/attach`
